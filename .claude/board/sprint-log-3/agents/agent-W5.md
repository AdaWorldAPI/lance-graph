## Decisions logged

1. **Manifest schema strictness: hard fail on missing required, soft on
   unknown top-level keys.** Required-field set is locked in the
   "Manifest schema" table in the spec. Unknown keys emit
   `cargo:warning` (forward-compat). Unknown nested keys under
   `stack_profile` are silently accepted (per-domain knobs vary).
2. **Generated files commit to git, not `.gitignore`.** Two reasons:
   `git blame` on `ogit_namespace.rs` shows when a slot was added;
   CI doesn't have to regen on every PR. The build script's
   `rerun-if-changed` directives keep the generated bytes honest.
3. **`inert_when_consumer_absent` detection is feature-flag based.**
   Each consumer crate exposes a `module-<name>` Cargo feature in
   `lance-graph-contract`. Build script reads `CARGO_FEATURE_MODULE_<NAME>`
   env vars and emits `#[cfg(feature = "module-<name>")]` gates around
   `ConsumerPointer::new::<...>()` calls in `seed_from_manifests()`.
   Rejected alternatives: scanning `workspace.members` (false positives)
   and `cargo metadata` (heavy build-dep, slow).
4. **Manifests live at workspace-root `modules/`**, not under
   `crates/lance-graph-contract/modules/`. They're workspace-level
   metadata visible to all crates and to non-Rust tooling (e.g., a
   future Python script that lists known consumers).
5. **`hubspo` placeholder ships now with `inert_when_consumer_absent: true`.**
   It's the regression gate for the W8 consumer-template dry-run: if
   shipping `hubspo-rs` later requires touching `lance-graph-contract`,
   the architecture has regressed.
6. **Canonical OGIT-G slot table locked in this spec:**
   DOLCE=0, MED=1 (reserved), HEALTHCARE=2, GOTHAM=3, SMB=4, FMA=5, CRM=6.
   Build script emits constants only for slots claimed by a manifest
   (slot 1 stays unbound until reclaimed by a follow-up).
7. **Generated files split into two:** `ogit_namespace.rs` (consts only)
   and `registry_seed.rs` (factory function). Keeps the constant table
   light enough to be a `pub use` target without dragging registry
   construction into every consumer crate.

## Dependency call-out

PR-E-1 depends on **PR-B-1 (W3)** landing first. The build script emits
code that references `ContextBundle` and `ConsumerPointer` types from
PR-B-1's contract surface. Without those types, the generated
`registry_seed.rs` won't compile.

PR-E-1 is a **sister change to PR-C-1 (W4)**: both touch
`crates/lance-graph-contract/src/consumer.rs`. PR-C-1 owns the trait
shape; PR-E-1 adds the `seed_from_manifests()` factory that consumes it.
Coordinate edits to avoid merge conflicts.

PR-F-1 (W6) reads manifests at runtime via `seed_from_manifests()` and
is a downstream consumer of this spec.
