# PACKED OWL + DOLCE SCHEMA BINARY FORMAT

**Version**: 0.1 (Initial)  
**Target size**: ≤ 50 KB for full OGIT (513 classes, 375 verbs, 856 attributes + characteristics + DOLCE)  
**Storage**: Content-addressable in Lance via Bgz tensor crate or raw bytes; keyed by 3-5 byte short pointer  
**Consumers**: lance-graph-owl-simd validator, MUL gate, polyglot planner, Investigation Agent, admin introspection tools

## 1. Design Goals

- L1 cache resident (entire schema in < 64 KB)
- O(1) or near-O(1) access for the hot paths:
  - Subclass test
  - Property domain/range check
  - Functional property multi-match detection
  - DOLCE Endurant/Perdurant classification
- Versioned & content-addressable (historical data always validates against the schema that existed when it was written)
- Extensible without breaking old readers (version field + reserved space)
- Generated by glue layer from OGIT TTL; consumed as read-only blob by SIMD kernels

## 2. Top-Level Layout (Little-Endian)

```
Offset   Size    Field
0        4       Magic (0x4F574C53 = "OWLS")
4        2       Format Version (current = 1)
6        2       Flags (bit 0 = has_DOLCE, bit 1 = has_equivalence_table, ...)
8        16      Source Content Hash (BLAKE3 or SHA-256 of canonicalized OGIT TTL)
24       2       Num Classes (C)
26       2       Num Properties (P)
28       4       Class Hierarchy Bitmap Offset
32       4       Property Characteristics Offset
36       4       DOLCE Category Bitmap Offset
40       4       Domain/Range Table Offset
44       4       Equivalence Table Offset (0 if none)
48       4       Annotation String Pool Offset
52       4       Reserved / Future
56       8       Schema Generation Timestamp (unix ns)
64+      ...     Variable sections (see below)
```

**Total header**: 64 bytes (generous for future growth while staying tiny).

## 3. Class Hierarchy Bitmap Section

**Purpose**: Fast `is_subclass_of(class_id, ancestor_id)` and transitive queries.

**Format options** (choose at compile/hydration time based on C):
- Dense bit matrix: C × C bits (for C ≤ 512 feasible: 513×513/8 ≈ 33 KB worst case — acceptable)
- Or RoaringBitmap / hierarchical bitmap for sparsity
- Or adjacency list + precomputed transitive closure (if we want planner hints)

**Recommended for v1**: Dense bit matrix stored row-major, bit-packed.  
AMX / ndarray matrix kernels can accelerate bulk operations or hydration-time transitive closure computation.

**Access**: `bitmap[class_id * stride + ancestor_id / 8] & (1 << (ancestor_id % 8))`

## 4. Property Characteristics Section

**Per-property record** (fixed size for speed):

```
Offset (relative)  Size  Field
0                  1     Characteristics Bitfield
                       Bit 0: Functional
                       Bit 1: InverseFunctional
                       Bit 2: Transitive
                       Bit 3: Symmetric
                       Bit 4: Asymmetric
                       Bit 5: Reflexive
                       Bit 6: Irreflexive
                       Bit 7: Reserved
1                  2     Domain Class ID (short internal ID)
3                  2     Range Class ID
5                  1     Cardinality Hint (0=unknown, 1=functional, 2=... )
6                  2     Inverse Property ID (0 = none)
8                  ...   Padding to 16-byte alignment for SIMD loads
```

**Total per property**: 16 bytes (padded).  
For 375 properties ≈ 6 KB.

**MUL fast path**: Load the 1-byte bitfield for a property; test bit 0 (Functional). If set and planner returned >1 candidate → veto.

## 5. DOLCE Category Bitmap

Simple bit vector: 1 bit per class.

```
Bit i = 0 → dolce:Endurant
Bit i = 1 → dolce:Perdurant
```

**Size**: (Num Classes + 7) / 8 bytes (≈ 64 bytes for 513 classes).

**Validator fast path**: Single load + bit test. Used by:
- Investigation agent (different counterfactual semantics)
- Planner (endurants can be mutated via interventions; perdurants are immutable records)

## 6. Domain / Range Table (optional dense form)

If not inlined in property records, a separate table for bulk checks.

For v1 we inline in the per-property record above.

## 7. Equivalence Table (for multi-vocabulary assimilation)

```
Num Equivalence Pairs (u16)
Then pairs of (ClassID_A, ClassID_B) — short internal IDs
```

Small (even 1000 pairs = 4 KB). Enables `owl:equivalentClass` / `sameAs` resolution in O(1) or log time via sorted table or perfect hash.

## 8. Annotation String Pool

Null-terminated UTF-8 strings for:
- `rdfs:label`
- `rdfs:comment`
- `dcterms:source`, `dcterms:creator`, etc.
- Original `ogit:*` names for debugging/audit

Offset table at the beginning of the section points into the pool.

Used by:
- Admin dashboards
- Audit log rendering
- Code generation for Action Types / DTOs

## 9. How ndarray / Bgz Integrate

- **At hydration time (glue layer)**:
  - ndarray AMX / BLASGraph kernels used to build the class hierarchy bitmap (matrix ops) and any transitive hints.
  - Bgz tensor crate can compress the final packed blob before writing to Lance (or store the raw bytes; Bgz for very large future schemas).

- **At runtime (validator)**:
  - Packed schema is memory-mapped or loaded into a small arena.
  - Hot loops use plain bitwise ops + the occasional ndarray SIMD kernel for batch validation of thousands of triples.
  - Bgz tensor used when the schema (or AwarenessColumn signatures) must be stored persistently inside Lance tables.

## 10. Versioning & Compatibility

- Format Version in header.
- Old readers must be able to validate data written with older schemas (they only need the schema that was current at write time — stored alongside the data via the content pointer).
- New optional sections are appended; offset fields allow skipping unknown sections.
- Breaking changes → bump major version + new magic or explicit migration path in glue layer.

## 11. Example Sizes (Full OGIT)

- Header: 64 B
- Class hierarchy (dense 513×513 bits): ~33 KB
- 375 properties × 16 B: ~6 KB
- DOLCE bitmap: 64 B
- Equivalence table (assume 200 pairs): 800 B
- Annotations (labels + comments): ~10–15 KB (string pool)
- **Total**: ~50–55 KB → comfortably under target with minor tuning (sparse bitmap or Roaring if needed).

## 12. Files & Code Locations

- Definition + (de)serializer: `lance-graph-owl-simd/src/packed_schema.rs`
- Compiler (from in-memory OWL model): `lance-graph-owl-simd/src/compiler.rs`
- ndarray kernel calls: `lance-graph-owl-simd/src/ndarray_kernels.rs` (thin wrapper)
- Bgz integration: via `bgz_tensor` crate re-export or direct dependency in the above

## 13. Validation Invariants (enforced by glue + compiler)

- All class/property IDs are dense [0 .. Num-1]
- No duplicate property characteristics for same (domain, verb) pair
- Functional + InverseFunctional only on properties that make semantic sense
- DOLCE bit consistent with explicit or heuristic classification
- Content hash in header matches the OGIT source that produced it

---

This format is deliberately simple and cache-friendly. The heavy lifting (AMX matrix work, vectorized cardinality) happens either at hydration (once) or in the SIMD validator kernels (hot path) using the primitives from AdaWorldAPI/ndarray.
